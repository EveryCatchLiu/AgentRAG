# 模块 2：自建检索 + 记忆

**复杂度：** 🔴 复杂 - 涉及文件处理、向量数据库、嵌入模型、检索集成，需要多步骤执行

## 2.1 Supabase Storage + 数据库 Schema

### 新表
- `files`: `id`, `user_id`, `filename`, `storage_path`, `status` (pending/processing/done/error), `total_chunks`, `created_at`
- `chunks`: `id`, `file_id`, `content`, `embedding (vector(1536))`, `chunk_index`, `created_at`

### Supabase Storage Bucket
- 创建 `documents` bucket，私有，RLS 策略

### RLS
- 所有表/策略：用户只能看到自己的文件和 chunks

### 验证
- [ ] 表创建成功，pgvector 扩展已启用
- [ ] Storage bucket 创建成功

## 2.2 文件导入后端

### API 端点
- `POST /api/files/upload` - 上传文件到 Supabase Storage + 创建记录
- `GET /api/files` - 获取用户的文件列表
- `DELETE /api/files/{id}` - 删除文件（级联删除 chunks）

### 文件上传
- 使用 Supabase Storage API
- 支持 `.txt` 格式（后续模块扩展格式）
- 文件存储路径：`{user_id}/{file_id}/{filename}`

### 验证

## 2.3 分块与嵌入管线

### 分块策略
- 按段落/句子分割，默认 1000 tokens/块，200 token 重叠
- 简单实现：按 `\n\n` 分段落，按字符数截断

### 嵌入生成
- 使用 DashScope 的 `text-embedding-v3` 模型（1536 维）
- 批量调用（每次 10 个 chunks）

### 存储
- 将 chunk 内容和 embedding 写入 `chunks` 表
- 使用 pgvector 的 `halfvec` 或 `vector` 类型

### 验证
- [ ] 上传文件后自动触发分块
- [ ] chunks 正确存储到数据库
- [ ] embedding 维度正确（1536）

## 2.4 检索工具

### 向量搜索
- 查询嵌入 → pgvector 相似度搜索 → 返回 Top-K chunks
- 使用 `<=>` 或 `distance` 运算符

### 后端工具函数
- `search_chunks(query: str, top_k: int = 5) → List[Chunk]`
- 流程：query → 嵌入 → pgvector 搜索 → 返回结果

### 验证
- [ ] 给定查询能返回相关的 chunks
- [ ] 搜索结果按相关性排序
- [ ] 不同查询返回不同结果

## 2.6 聊天集成（检索注入）

### 消息流程
1. 用户发送消息
2. 后端检索相关 chunks
3. 将 chunks 作为 context 注入系统提示
4. 调用 LLM 生成回复
5. 流式返回

### 系统提示
- "你是一个 RAG 助手。基于以下文档片段回答用户问题。如果片段中没有答案，请说明你无法找到相关信息。"

### 验证
- [ ] 聊天能引用文档内容回答
- [ ] 当文档中没有答案时明确说明
