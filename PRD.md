# AgentRAG  - 产品需求文件

## 我们要构建什么

一个具有两个界面的 RAG 应用程序：

1. **聊天**（默认视图）- 带有检索增强回复的线程对话
2. **文件导入** - 手动上传文件、追踪处理进度、管理文件

这**不是**带有连接器的自动化管道。文件通过拖放方式手动上传。通过环境变量进行配置，没有管理界面。

## 目标用户

想要使用 AI 编程工具（Claude Code、Cursor 等）构建生产级 RAG 系统的技术人员。他们不需要懂 Python 或 React - 那是 AI 的工作。

**他们需要理解的：**

- 深入理解 RAG 概念（分块、嵌入、检索、重排序）
- 代码库结构（什么在哪里，各部分如何连接）
- 如何指导 AI 构建他们需要的东西
- 如何指导 AI 修复出现的问题

## 技术栈


| 层次         | 选择                                               |
| ---------- | ------------------------------------------------ |
| 前端         | React + TypeScript + Vite + Tailwind + shadcn/ui |
| 后端         | Python + FastAPI                                 |
| 数据库        | Supabase（Postgres + pgvector + 认证 + 存储 + 实时）     |
| LLM（模块 1）  | OpenAI Responses API（托管线程 + 文件搜索）                |
| LLM（模块 2+） | 任何 OpenAI 兼容端点（OpenRouter、Ollama、LM Studio 等）    |
| 可观测性       | LangSmith                                        |


## 约束条件

- 不使用 LLM 框架 - 直接使用 OpenAI SDK 的标准 Chat Completions API（OpenAI 兼容），使用 Pydantic 进行结构化输出
- 所有表都需要行级安全性 - 用户只能看到自己的数据
- 通过 SSE 进行流式聊天
- 通过 Supabase Realtime 进行导入状态更新

## 模块 1：应用外壳 + 可观测性

**构建：** 认证、聊天 UI、OpenAI Responses API（管理线程 + 文件搜索）、LangSmith 追踪

**学习：** 什么是 RAG、为什么存在托管 RAG、它的局限性（OpenAI 处理记忆和检索 - 黑盒）

**备注：** Responses API 是 OpenAI 特有的。它提供托管线程和内建文件搜索，但将你锁定在 OpenAI。模块 2 过渡到标准 Chat Completions API 以实现提供商灵活性。

---

## 架构决策：模块 1 → 模块 2 过渡

在模块 1 结束时，你有一个使用 OpenAI **Responses API** 的工作聊天应用——这是一个由 OpenAI 处理线程、记忆和文件搜索的托管解决方案。在模块 2 中，你切换到标准 **Chat Completions API**，以支持任何 OpenAI 兼容的提供商（OpenRouter、Ollama、LM Studio 等）。

**你需要做的决定：** 如何处理 Responses API 代码？以下是两种常见方法，但你不限于这些——如果对你的用例有意义，可以提出自己的方案。


| 选项         | 方法                                            | 优点              | 缺点                     |
| ---------- | --------------------------------------------- | --------------- | ---------------------- |
| **A：替换**   | 完全移除 Responses API 代码，基于 Chat Completions 重建  | 代码库干净、单一模式、更易维护 | 失去使用 OpenAI 托管 RAG 的能力 |
| **B：双重支持** | 保留 Responses API 与 Chat Completions 并存，可按请求配置 | 灵活使用任一方法，可并列比较  | 代码库更复杂，需理解两种模式         |


没有正确答案——这是构建生产系统时你将面临的真实架构选择。

**在视频中，我选择了选项 A**——完全从代码库中移除 Responses API 代码以及数据库中任何相关的模式。这使事情保持简单，专注于之后的 OpenAI 兼容 Chat Completions 模式。

**这是一堂关于如何引导 Claude Code 的课**：你需要清楚地传达你的决定，并引导 AI 正确实施。明确说明你想要移除、重构或保留什么。

## 模块 2：自建检索 + 记忆

**前提条件：** 完成上述架构决策。

**构建：** 导入 UI、文件存储、分块 → 嵌入 → pgvector、检索工具、Chat Completions API 集成（OpenRouter/Ollama/LM Studio）、聊天历史存储（无状态 API - 现在由你管理记忆）、实时导入状态

**学习：** 分块、嵌入、向量搜索、工具调用、相关性阈值、管理对话历史、**引导 AI 代理进行架构重构**

## 模块 5：多格式支持

**构建：** 检测全文件场景、生成具有自己工具的隔离子代理、UI 中的嵌套工具调用显示、展示主代理和子代理的推理过程
